## 1. `stopAllClipsBestEffort` (new, `AbletonAdapter.ts:62-85`) — command identity and guard — PASS

```
async function stopAllClipsBestEffort() {
  if (!tracks?.length) {
    Logger.warn('Skipping crash-exit stop_all_clips: Ableton tracks not yet loaded');
    return;
  }
  try {
    await Promise.all(
      tracks
        .slice(0, 4)
        .map((track, pillar) =>
          track
            .sendCommand('stop_all_clips')
            .catch((err) => Logger.error(err, `Error stopping clips on pillar ${pillar + 1} during crash exit`)),
        ),
    );
  } catch (err) {
    Logger.error(err, 'Error attempting best-effort stop_all_clips during crash exit');
  }
}
```

Confirmed `Track` (`ableton-js/ns/track.js`) does not override `sendCommand` — it inherits `Namespace.prototype.sendCommand` unmodified (`__extends(Track, _super)` against `_1.Namespace`; grepped the whole file for a `Track.prototype.sendCommand` override and found none). So all four call sites resolve to the identical method with identical arguments — there is no new command, no new argument shape, nothing musically novel here.

## 2. Parallel vs. sequential dispatch across the 4 tracks — independently verified against `ableton-js` transport source — PASS, no ordering hazard

This is the question I was specifically asked not to take on trust, so I traced it through the actual installed library rather than the implementer's summary of it:

- `Namespace.prototype.sendCommand(name, args, etag, timeout)` (`ableton-js/ns/index.js:97-103`) forwards to `Ableton.prototype.sendCommand({ns: this.ns, nsid: this.nsid, name, args, etag}, timeout)`. Each `Track` instance carries its own `nsid` (the track's own Live object id) — so a `stop_all_clips` sent to `tracks[0]` and one sent to `tracks[1]` are addressed to different objects in the payload itself, not disambiguated by send order.
- `Ableton.prototype.sendCommand` (`ableton-js/index.js:451-487`) generates a fresh `uuid` per call (`msgId`), stores `{res, rej, clearTimeout}` in `this.msgMap` — a `Map`, not a single in-flight slot — keyed by that `msgId`, then calls `this.sendRaw(msg)`. Response correlation on the way back is by `msgId`/`uuid`, not by arrival order (confirmed in the WOW-014 audio-ableton-reviewer note's independent read of `handleUncompressedMessage`, unchanged in this PR). This means the client-side promise bookkeeping is inherently safe for concurrent, out-of-order-completing commands; nothing here assumes commands complete in the order they were sent.
- `Ableton.prototype.sendRaw` (`index.js:614-631`) deflates the JSON payload and sends it over a `dgram` UDP4 socket (`this.client.send(chunk, 0, chunk.length, this.serverPort, "127.0.0.1")`) to localhost. A `stop_all_clips` command carries no `args`, so its serialized payload is trivially under the 7500-byte chunking threshold — one UDP datagram per command, sent synchronously the moment `.map()`'s callback runs (before any `await` in the caller yields). So `Promise.all(tracks.slice(0,4).map(...))` really does fire all 4 datagrams back-to-back in the same tick, exactly as the implementer's note claims — independently confirmed, not assumed.
- **Ordering-dependency check:** nothing in this repo's docs (`ABLETON_INTEGRATION.md`, `DATA_MODEL.md`) or in `AbletonAdapter.ts` documents or relies on `stop_all_clips` needing to happen in a specific cross-track order. `TRIGGER_ORDER` and `KEY_LEADER_ORDER` govern which clip becomes phrase leader and playback/key-timing relationships between _currently playing_ clips — they have no bearing on a blanket "stop everything on this track" broadcast, and `stopAllClipsBestEffort` doesn't read or write `TRIGGER_ORDER`, `KEY_LEADER_ORDER`, `phraseLeader`, `masterKey`, or `playingClips`/`queuedClips` at all (confirmed by reading the full function body — it touches only `tracks` and `Logger`). Each track's `stop_all_clips` is self-contained: stopping pillar 0's clips has no dependency on, and no effect on, pillar 1-3's clips. I found no `ableton-js` or Remote-Script-side mechanism (visible in this repo) that would make concurrent per-track commands to _different_ `nsid`s race unsafely — the closest thing to a shared resource is the single UDP client socket, and OS-level UDP send calls for four small back-to-back datagrams on loopback do not corrupt or reorder each other's _payloads_ (each `client.send()` call is a complete, independent datagram write).
- **Comparison to `handleTimeout`'s existing sequential loop:** that loop's `await`-in-series shape guarantees _send_ order (track 0 before track 1, etc.) but, being UDP, has never guaranteed _delivery/execution_ order either — this PR does not remove any ordering guarantee that previously existed for correctness, because none existed beyond "all 4 get sent." Sequential-vs-parallel changes wall-clock timing (parallel is faster, and required to fit the reviewer's 1-2s bound per the implementer's own investigation in their note — a sequential loop could cost up to ~4× the per-command 2000ms timeout, unacceptable for this bound), not command semantics.
- **Conclusion:** parallel dispatch across tracks introduces no musical-correctness risk beyond what already existed in the transport. This matches the implementer's claim; I verified it against the library source rather than accepting the summary.

## 3. `crashExit` / `backend/index.ts:12-41` — bound guarantee and re-entrancy — PASS, fully traced

- **Bound guarantee:** `await Promise.race([AbletonAdapter.stopAllClipsBestEffort(), new Promise((resolve) => setTimeout(resolve, CRASH_EXIT_STOP_TIMEOUT_MS))])`. The timer promise unconditionally calls `resolve` after 1500ms — it has no rejection path — so `Promise.race` is guaranteed to settle within ~1500ms of `crashExit` starting, regardless of what `stopAllClipsBestEffort()` does (hang is not possible for it either, since each `sendCommand` has its own independent ~2000ms internal timeout per point 2, and every one of the 4 is individually `.catch()`-guarded so `Promise.all` inside it cannot itself hang or reject). `process.exit(1)` is the very next statement after the `await`, unconditionally reached once the race settles.
- **What if `stopAllClipsBestEffort()` itself somehow throws/rejects** (e.g., a hypothetical broken `Logger`)? Traced this explicitly: a rejection from `stopAllClipsBestEffort()` would make the `Promise.race` (and therefore the `await` in `crashExit`) throw, which propagates out of `crashExit`'s own async body — but both call sites wrap the call in `.catch(() => process.exit(1))` (`index.ts:36`, `:40`). This is the one real safety net for an unforeseen failure in the new code, and it is present at both registration sites. I traced this rather than assuming the comment describing it ("if crashExit ever does throw... process.exit(1) still fires immediately") was accurate, and confirmed it is.
- **Re-entrancy (`isCrashExiting`):** set to `true` before the stop attempt begins. A second `unhandledRejection`/`uncaughtException` firing mid-shutdown logs its own error (so the second failure's cause isn't lost) and returns without re-attempting a stop or racing a second `process.exit()`. Musically this is safe because the _first_ invocation's `stopAllClipsBestEffort()` already targets all 4 pillars unconditionally — it is not scoped to whichever error triggered it — so a second, different crash arriving in the same ~1.5s window doesn't leave any pillar uncovered by virtue of losing the race to be first; the first attempt was already comprehensive.
- **Logging parity:** `Logger.error(err, message)` inside `crashExit` is called with the same arguments (`err`/`reason`, and the same two message strings `'Unhandled promise rejection'` / `'Uncaught exception'`) that WOW-014's handlers called directly — confirmed via the diff; only the location moved, not the shape or content of the log call.
- **`main()`'s startup-failure `.catch()` (`index.ts:70-73`) is untouched** — still logs and calls `process.exit(1)` directly, does not call `crashExit`. See non-blocking observation below.

## 7. Degraded-outcome judgment: partial stop if the 1500ms bound fires mid-flight — acceptable for a crash path

Explicitly asked to render a judgment call here. If the 1500ms timer wins the race before all 4 `sendCommand` promises have settled (e.g., Ableton is slow to respond, or a network hiccup on loopback), the outcome is "some pillars got a `stop_all_clips` acknowledgement, others may not have" — and `process.exit(1)` fires regardless. In my judgment this is an acceptable, correctly-scoped degraded outcome for a crash-exit path specifically:

- It is strictly better than the pre-WOW-034 status quo (zero pillars ever attempted), which is the only baseline this ticket was asked to improve on.
- The alternative — waiting for full confirmation from all 4 tracks before exiting — reintroduces exactly the "unbounded await inside a crash handler could hang the exit" risk the original hardware-safety-reviewer explicitly warned against and declined to accept.
- The commands are fired in parallel, not staggered, so in the by-far-most-likely case (Ableton alive and responsive, which is the normal operating condition this installation runs under) all 4 settle within a handful of milliseconds, well inside the 1500ms bound — partial coverage is the exception case, not the expected case.
- Nothing about a partial stop leaves Ableton in a state worse than "still playing," which is also the outcome of every crash before this PR existed. There's no scenario where parallel dispatch or the bound firing mid-flight causes a track that _was_ stopped to resume, retrigger, or transpose unexpectedly — the only failure mode is "didn't get silenced," not "did something musically wrong."
