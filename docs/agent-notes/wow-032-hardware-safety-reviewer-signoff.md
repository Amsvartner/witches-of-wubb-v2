## 2. Startup-timeout exit behavior — PASS

- **The timeout mechanism is pre-existing `ableton-js` library surface, not new logic this PR wrote.** Confirmed by reading `backend/node_modules/ableton-js/index.d.ts:82` (`start(timeoutMs?: number): Promise<void>`, doc comment: "If set, the function will throw an error if it can't establish a connection in the given time") and the implementation in `index.js:189-307`: when `timeoutMs` is truthy, it does `Promise.race([connection, timeout])` where `timeout` is `new Promise((_, rej) => setTimeout(() => rej("Connection timed out."), timeoutMs))`. Before this PR, `ableton.start()` was called with no argument — per this same code (`if (!timeoutMs) return [3/*break*/,7]`), that means an **unbounded** wait, exactly the hang-forever bug the ticket describes. This PR's only change is supplying a value; it does not touch the library's timeout implementation.
- **Exit path:** the re-thrown `Error` in `AbletonAdapter.ts`'s new `catch` block propagates out of `startAbleton()` (nothing catches it locally) to `backend/index.ts:23`'s `await AbletonAdapter.startAbleton()` — the first line of `main()` — and from there to `main().catch((err) => { Logger.error(err, 'Fatal error during backend startup'); process.exit(1); })` at `index.ts:49-52`. Read `index.ts` in full (zero diff in this PR, confirmed via `git diff --stat`): this is WOW-014's existing, already-reviewed exit path. **No second exit mechanism exists in this diff.**
- **Ordering, confirmed by direct read, not by trusting the PR description:** `getTracksAndClips()` (which registers the four `playing_slot_index` listeners — the only place any pillar/clip/track state begins to exist) and `getTrackVolumes()` are the last two statements in `startAbleton()`, strictly after the try/catch and after `logConnectedRemoteScriptVersion()`. A timeout means neither runs. `main()` also never reaches the `socketio.Server`/`nodeOSC.Server` construction (both after the `startAbleton()` await), so a startup timeout means no websocket server, no OSC server, no listener, nothing playing, nothing queued, no pillar-facing state of any kind — the process exits before it could leave anything in a stuck playing/mid-transition state, because there is nothing yet to get stuck. This matches the task's framing and the ticket's own claim; I traced the control flow directly to confirm it rather than accepting it as asserted.
- **Reconnect-storm check:** no retry loop wraps `startAbleton()` or `main()` anywhere in this codebase — a timeout produces exactly one `process.exit(1)`, not repeated connection attempts. Cross-checked `docs/DECISIONS_NEEDED.md:18-30` (WOW-014's still-open crash-supervision entry, options nodemon `--exitcrash`/pm2/launchd/systemd all undecided): **no automatic process supervisor exists yet in this repo**, so today a startup timeout cannot trigger a rapid reconnect-storm against real hardware — nothing currently auto-restarts the process at all. This is unchanged by WOW-032 and is exactly the gap WOW-014 already flagged; noting it here only because the ticket explicitly asked this PR to "coordinate" with it. When that supervision decision is eventually implemented, this PR's actionable log message (naming the three causes) is the mechanism that keeps a future crash-loop diagnosable rather than silent — worth re-confirming backoff behavior at that time, but that's a future ticket's concern, not a gap in this one.

## 3. Post-connect version check is read-only — PASS

- Confirmed at the library-type level, not just from the PR's prose: `backend/node_modules/ableton-js/ns/internal.d.ts` declares `GettableProperties { version: string; ping: boolean }` and `SettableProperties {}` (empty) — `version` has no setter in the library's own contract.
- Confirmed at the implementation level: `backend/node_modules/ableton-js/ns/index.js` — `Namespace.prototype.get(prop)` calls `this.ableton.getProp(...)`; `Namespace.prototype.set(prop, value)` calls `this.ableton.setProp(...)` — disjoint code paths. `ableton.internal.get('version')` can only ever reach `getProp`. This is a read, independently verified at the library level.
- `logConnectedRemoteScriptVersion()` (`AbletonAdapter.ts:104-122`) wraps the RPC call itself in try/catch; a failure there is caught, logged via `Logger.warn`, and does not propagate — `getTracksAndClips()`/`getTrackVolumes()` still run afterward. Confirmed by reading the function body directly.
- **Minor, non-blocking observation:** inside both `logConnectedRemoteScriptVersion()` and `checkRemoteScriptVersionPreflight()`, the line `parseRemoteScriptVersion(getShippedRemoteScriptVersionPath())` sits _outside_ any try/catch (only `parseRemoteScriptVersion`'s own internals, and separately the `ableton.internal.get('version')` call, are wrapped). `getShippedRemoteScriptVersionPath()` calls `require.resolve('ableton-js/package.json')`, which could in principle throw on a module-resolution failure. In practice this is not a real exposure: `AbletonAdapter.ts` already does `import { Ableton } from 'ableton-js'` at module scope (line 4 — pre-existing, not part of this diff), so if that resolution could ever fail, the process would already have failed to load this module long before `startAbleton()` runs. Even in the hypothetical worst case, an uncaught throw here fails exactly the same direction as a timeout — propagates to `main().catch()` → `process.exit(1)`, and only ever before `getTracksAndClips()`/`getTrackVolumes()` (pre-flight case) or after a connect but still before them (post-connect case). No live command is ever sent and no pillar state is ever created either way. Not a blocker; flagging because tracing every unguarded throw is the job.

## 6. `.env` change — PASS, confirmed inert without operator action

- Both new vars are added as **commented-out** lines (`# ABLETON_START_TIMEOUT_MS=45000`, `# ABLETON_REMOTE_SCRIPT_VERSION_PATH=...`). `dotenv` does not parse commented lines, so `process.env.ABLETON_START_TIMEOUT_MS`/`process.env.ABLETON_REMOTE_SCRIPT_VERSION_PATH` remain `undefined` for any operator who pulls this PR without manually editing `.env`. Verified this against the actual consuming expressions, not assumed: `Number(process.env.ABLETON_START_TIMEOUT_MS) || 45000` (note: `Number(undefined)` is `NaN`, which is falsy, so this correctly falls through to `45000`) and `process.env.ABLETON_REMOTE_SCRIPT_VERSION_PATH || DEFAULT_REMOTE_SCRIPT_VERSION_PATH`.
- Side note, not a concern: `Number("0") || 45000` would also fall through to `45000` — meaning there is no way to configure "0 timeout"/opt back into the old unbounded-wait behavior via this var. Given the entire point of the ticket is to guarantee a bounded startup wait, this is the safe direction to fail in, not a footgun.
- The one real, intentional behavior change that applies **regardless** of whether `.env` is edited — `ableton.start()` goes from an unbounded wait to a bounded 45-second wait by default — is disclosed in the PR body, the README addition, and the ticket itself, and is a net safety improvement (previously an indefinite hang with zero diagnostic feedback was possible; now a bounded, logged, actionable failure). This is the only behavior shift; no operator is exposed to an undisclosed change merely from pulling this PR.
- No other `.env` key touched — `OSC_SERVER_PORT`, `LIGHTING_SERVER_ADDRESS`, `LIGHTING_SERVER_PORT`, `VITE_WS_SERVER_ADDRESS` all present, unchanged (diff only appends after the existing four lines).
