## 2. New crash-exit behavior (`process.on('unhandledRejection'/'uncaughtException')` → `process.exit(1)`) — reviewed, no blocker, non-blocking observation below

`backend/index.ts:12-20` registers both handlers at module-load time, before `main()` runs (`index.ts:49-52`); both log via `Logger.error` then call `process.exit(1)`. This is the one genuinely new runtime behavior in the PR (everything else is additive error handling around calls that already existed).

Analysis performed:

- **Does this make full-process crashes more likely than on `main`?** No — net effect is the opposite. Node has terminated processes on uncaught synchronous exceptions unconditionally since forever, and on unhandled promise rejections by default since Node 15 (no `--unhandled-rejections` flag or `engines`/`.nvmrc` pin exists in this repo to opt out — confirmed via `find` for `.nvmrc`/`.node-version` and grep for `engines` in both `package.json` files: none found). So on any reasonably current Node runtime, the crash-trigger bugs this ticket fixes (e.g. the `phraseLeader` throw) were **already** fatal to the whole process before this PR — this PR mainly replaces Node's raw stderr dump with a structured pino log line before the same exit. Meanwhile, the bulk of this PR's changes are `.catch()` handlers newly attached at dozens of call sites; a promise with a `.catch()` attached synchronously (as every site in this diff does) never fires `unhandledRejection` at all. So most of the errors that would have crashed the process on `main` are now caught and logged locally and **do not reach** the new process-level handlers or exit the process. The new handlers are a backstop for whatever's left, not a new source of exits.
- **Could a crash-then-exit during active playback leave anything in an unsafe state?** Traced what happens to Ableton on any process exit, old or new: Ableton Live is a separate application/process; nothing in `main` or in this PR sends a stop/silence command to Ableton as a consequence of the Node process exiting. `handleTimeout()` (`AbletonAdapter.ts:54-59`) is the only routine in the codebase that stops all four tracks (`for (let i = 0; i < 4; i++) await tracks[i].sendCommand('stop_all_clips')`), and it is wired only to the 3-minute idle-timeout timer, not to either new `process.on(...)` handler — this PR does not connect them, and was never asked to. So: if the backend dies (crash-exit either way), whatever is currently playing/looping on each pillar continues at whatever volume/LED-color was last set, indefinitely, until a human notices and restarts the backend (and, per the new `docs/DECISIONS_NEEDED.md` entry, nothing currently auto-restarts it either). **This exact gap exists identically on `main`** — an unhandled rejection there already kills the process with the same "nothing tells Ableton to stop" outcome. This PR does not introduce, worsen, or (within its stated scope) fix this; it is pre-existing.
- **Is this already covered by the supervisor discussion in `docs/DECISIONS_NEEDED.md`?** Partially. The new entry (added by this PR, `docs/DECISIONS_NEEDED.md:18-37`) frames the problem as "nothing restarts the process automatically after a clean crash-exit" and proposes `nodemon --exitcrash` / pm2 / launchd-systemd — i.e. it addresses **recovery speed**. It does **not** address **what Ableton does while the backend is down** (whether the exit handlers should attempt a bounded, best-effort "stop all 4 tracks" before calling `process.exit(1)`). These are related but distinct questions; the entry as written does not cover the second one.

**Non-blocking observation (major, worth a follow-up ticket, not a blocker on this PR):** Neither `main` nor this PR ever silences Ableton on process exit — a backend crash (via either the old implicit path or these new explicit handlers) leaves audio playing/looping on all active pillars indefinitely until a human restarts the process. This PR's own stop condition correctly kept "graceful shutdown" out of scope (attempting more Ableton RPCs from inside an `uncaughtException`/`unhandledRejection` handler is itself risky — the very state that caused the crash may make `tracks`/`ableton` unusable, and an unbounded await here could hang the exit instead of failing fast), so I am not blocking on this. Recommend: a follow-up ticket to add a _bounded_ (e.g. 1-2s timeout, wrapped in its own try/catch, never delaying `process.exit(1)` beyond that bound) best-effort "stop all clips on all 4 pillars" attempt inside the two new handlers before they exit, and an addition to the existing `docs/DECISIONS_NEEDED.md` crash-supervision entry noting that "what happens to live audio during the down-window" is a separate open question from "how fast do we come back up." This pairs naturally with WOW-032 (already flagged in the tickets doc as needing to "coordinate with WOW-014's Decision-needed entry" on process-exit semantics).

## 3. `playing_slot_index` listener: `throw` → `Logger.error` + `return` — PASS, minor non-blocking observation

Traced `playingClips[pillar]`/`stoppingClips[pillar]` immediately before/after this exact code path on both branches, for the missing-clip-metadata case (Ableton reports a new `clipSlotIndex`, `clipName` resolves, but `MusicDatabaseService.clipNameToInfoMap[...]` misses — e.g. CSV/Ableton clip-name mismatch):

- **Control-flow position is identical on both branches.** On `main` (`AbletonAdapter.ts:210` area) and on head (`AbletonAdapter.ts:311-324`), the `if (!clipMetadata) {...}` check sits in the exact same place: after `Logger.info(...)`, before the "transpose previous clip back to key 0" block, and before `playingClips[pillar] = { ...clipInfo, clip }`. Confirmed by diffing the surrounding ~70-line block statement-by-statement (ignoring indentation from the new `try{}` wrap) — every line of business logic is unchanged; only `throw new Error(...)` became `Logger.error(...); return;`, `setTrackVolume(...)`/`addPhraseLeader(...)` got `.catch()`, and the whole callback got a `try/catch` + the outer `.addListener(...)` got a `.catch()`.
- **State before this line, on both branches:** whatever `playingClips[pillar]`/`stoppingClips[pillar]` held from the _previous_ event (unrelated to this invocation).
- **State after this line, on `main`:** moot — the `throw` is inside an `async` callback that nothing awaits when `ableton-js` invokes it (confirmed: `track.addListener(...)` itself was never awaited/`.then`/`.catch`-ed on `main` either), so this was **always** an unhandled rejection, which (per point 2's analysis) already crashes the whole process on any current Node runtime. `playingClips[pillar]` is never reached/updated by this invocation, but that's irrelevant because the entire process — and therefore every pillar's state, not just this one — goes down.
- **State after this line, on head:** the function returns early; `playingClips[pillar]` is **not** updated to reflect the new clip Ableton is actually now playing (same as `main`, this assignment is below the check and unreached), but the **process keeps running**. This means `playingClips[pillar]` can be briefly stale/wrong relative to Ableton's actual audio (still showing the previous clip, or `null`) for as long as the mismatched clip keeps playing. Consequence: the currently-playing event (`clip_started`/`clip_playing`) is never emitted for the unrecognized clip, so the UI and the lighting server never learn about it — a pillar's LED could show a stale category color while a different, unrecognized clip is audibly playing on it.
- **Does this self-heal, or stay stuck indefinitely?** It self-heals on the next tag-departed event for that pillar. `stopOrRemoveClipFromQueue` (`AbletonAdapter.ts:196-257`, unchanged by this PR beyond `.catch()`s) has a pre-existing fallback: when a clip is neither recognized as playing nor queued (`!isClipPlaying && !isClipQueued`, true here because `playingClips[pillar]` never got the mismatched clip's name), it still sends `tracks[pillar].sendCommand('stop_all_clips')` "just in case" and emits `clip_stopped` (`AbletonAdapter.ts:250-255`) — this logic is untouched by this PR and already existed on `main`. So the audio does get stopped and the UI/LEDs do get corrected once the RFID tag is removed; the window of staleness is bounded by however long the object stays on the pillar, not indefinite.
- **Net assessment:** this is a pre-existing latent inconsistency (the `!clipMetadata` gate's position relative to the `playingClips` write was already there on `main`), not a new one introduced by this PR. What changed is that the system can now _reach and continue past_ this state instead of crashing outright — which, system-wide, is strictly safer (one pillar's display briefly stale and self-correcting vs. all four pillars, the UI, and RFID responsiveness going dark until a human restarts the process). Filing as **minor, non-blocking**: worth a follow-up ticket (natural pairing with WOW-017's pillar/error-context logging and WOW-023's CSV/backend key-normalization-divergence tracking) to have this branch also explicitly emit a `clip_unqueued`/warning-level UI event so the operator display doesn't silently lag, rather than relying solely on the next departed-tag event. Not a hardware-command hazard: no strobe, no volume change, no extra live command — if anything, fewer commands fire on this path than the happy path.
