## Findings

**1. [Required — fix or record explicit human waiver before merge] Non-space whitespace regression band.** `backend/util/ClipNameUtil.ts:2` strips only U+0020 and `*`; the replaced `.trim()` stripped _all_ whitespace (tab, CR/LF, non-breaking space U+00A0, BOM U+FEFF, …). Concrete regression, probe-verified: a Live clip named `"Wicked Casting\t"` (or with a trailing non-breaking space, a realistic copy-paste artifact) **matched CSV `"Wicked Casting"` on `main`** via trim, and **silently fails to match on this branch** — `FindAllClipsInLoop` returns `[]`, the clip can never be queued or fired, no diagnostic beyond the existing "No clip found" warn. This is a direct counterexample to the branch's own safety claim ("the helper is a strict superset of `.trim()`" — implementation note, and the test comment in `backend/util/test/ClipNameUtil.test.ts`); that claim holds only for U+0020. The prior review round explicitly raised non-breaking spaces to the human, and the human's answer ("spacing") most naturally covers them. Recommended fix: change the pattern to `/[*\s]/g` (JS `\s` includes U+00A0 and U+FEFF) plus a tab/nbsp test case. Probe-verified strictly safe: `[*\s]`-stripping is a superset of both `.trim()` and the current `[* ]`-stripping, so nothing that matches on `main` **or** on this branch stops matching — it can only widen. Not a block because the mainline decision is correctly implemented, the trigger characters are hard to type in Live's rename field (paste-only), and the fix is mechanical and inside this ticket's allowed files. **Resolved at `53960ee` — see addendum.**

**2. [Minor — TBD, constraint to confirm or fast-follow] Intra-block decoration variance vs raw comparisons in the `playing_slot_index` listener.** `FindAllClipsInLoop` now groups decoration variants into one loop block, but `backend/adapter/AbletonAdapter.ts:468` and `:492` still compare **raw** names. If slots _within_ one loop block carry differing decoration (e.g. `"Verse"`, `"Verse *"` as consecutive slots of one song), a slot advance takes the new-clip path: line 468 de-transposes the still-playing block to pitch 0 (audible key clash under key lock — and nothing re-transposes it, since transposition happens in `queueClip`), and line 492 emits `clip_started` instead of `clip_playing` (timeout reset + volume reset to 0.6). Whether real blocks vary decoration per slot is not determinable from the repo — **TBD, human to confirm**. If uniform-decoration-within-a-block holds, this finding is moot; if not, normalize these two comparisons in a fast-follow. Not a regression vs `main` (these lines are untouched, and on `main` such a set misbehaved differently anyway), but "safe to use asterisks" is only fully delivered under the uniform-decoration assumption. (`:514`'s raw comparison reviewed too: identity check on same-provenance strings — fine as-is.)

**3. [Minor — intended consequence, needs documenting] Conflation is now the contract.** Two genuinely different Live clips whose names differ only by spaces/asterisks now resolve to one — `MemoizedClipIndex` (`AbletonAdapter.ts:214-222`) returns the _earliest_ normalized match, so a decoration-variant clip in a lower slot silently wins. Acceptable and intended per the decision, but it makes two things hard naming-convention rules for the Live set: (a) distinct clips must differ by more than decoration; (b) loop blocks must remain contiguous runs of same-normalized-name slots — the 20-slot `findLastIndex` slice (`:241-247`) still includes any interleaved non-matching slots in the returned block (pre-existing shape, broader trigger now), and those would be wrongly transposed under key lock.

That file is outside this ticket's allowed list, so this is a follow-up docs pass (fold in finding 3's convention rules while there), not a defect of this diff.

**5. [Nit] No adapter-level test exercises the new matching.** `MemoizedClipIndex`/`FindAllClipsInLoop` have no test seam (pre-existing, same situation WOW-021 faced on this file); confidence rests on the helper's unit tests plus hand-trace.
