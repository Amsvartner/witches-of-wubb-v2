**2. [Minor, fixed] `docs/HARDWARE_INTEGRATION.md:24` and `docs/DECISIONS_NEEDED.md:70` both described the unknown-IP handling in present tense ("logged as an unknown-IP warning, WOW-017") as though it's already active.** Correct catch: this PR's branch stacks on `feat/wow-029-wifi-reconnect` → `feat/wow-028-wifi-secrets`, not on WOW-017's chain, and `backend/event/IncomingEvents.ts` is confirmed byte-identical to `main` on this branch — so WOW-017's actual guard doesn't exist yet from this branch's point of view; today an unmapped IP is silently swallowed by a generic `catch (err) { Logger.error('Errored trying find track from RFID tag'); }` with no IP or unknown-pillar framing. **3. [Minor, addressed via docs, not re-escalated] `secrets.h.example`'s shipped `PILLAR_IP` default (`192.168.0.101`, a real Pillar-0 address) risks an IP collision if more than one physical device is ever flashed without editing it — a different, network-level failure mode than the old ".52 matches nothing" default's silent-drop behavior.** This is a genuinely new angle the mandatory hardware-safety-reviewer sign-off (already recorded, APPROVE) didn't explicitly discuss. Considered looping back to hardware-safety-reviewer for a narrow follow-up question (this run's established pattern for a gap surfacing after an initial sign-off, e.g. WOW-018's pitch_coarse follow-up) but judged it disproportionate here: the finding doesn't request or need any code change, only a documentation clarification, and the underlying risk (a human skipping the flashing checklist's per-device verification step) is the same category of pre-existing human-error risk the ticket itself frames as inherent to "four devices require four undocumented hand edits" — this PR's job was making that mistake _diagnosable_, not eliminating human error entirely. Added an explicit callout to `docs/HARDWARE_INTEGRATION.md`'s "Pillars" section naming the trade-off directly and confirming there's no code-level guard against it (a compile-time cross-device uniqueness check isn't possible per-sketch; a runtime IP-conflict detector is out of this ticket's scope). If a future reviewer or the human disagrees with this proportionality call, the fix is cheap (an obviously-invalid placeholder default, trading back to the old "silently drops" failure mode) and can be revisited.
