## Findings against the six review points

**1. Timing subtlety in the new "drops any still-queued clip" test — sound.**
Read `queueClip` (`sim/core/simulator.ts:239-263`) and `schedulePhraseTrigger` (`:268-274`) directly: any clip that queues (i.e., `silence` is false) unconditionally arms `phraseTimerId = setTimeout(triggerQueuedClips, phraseLengthMs)`, independent of the idle-timeout timer. The suite's `PHRASE_LENGTH_MS = 1000` (confirmed at `sim/test/simulator.test.ts:29`), and the main `simulator` instance in `beforeEach` (`:39-43`) does **not** override `timeoutMs`/`timeoutWarningMs`, so it runs the real default `TIMEOUT_IN_MILISECONDS` (3 minutes). Under that default, a queued clip auto-fires at t=1000ms — over 100x sooner than a default timeout could ever fire — so the "still queued at timeout" state is genuinely unobservable on the default-configured instance, confirming a dedicated short-timeout instance was necessary, not just convenient.

I hand-traced the new test's actual timer schedule: both `handleNewTag` calls happen synchronously at simulated t=0 (each internal `emit(..., true)` call re-arms `restartTimeoutTimer()` from t=0, since fake timers don't advance on synchronous code), leaving `timeoutWarningId` at t=400, `timeoutId` at t=500, and `phraseTimerId` at t=1000 all queued from the same t=0 baseline. `vi.advanceTimersByTime(500)` lands exactly on `timeoutId`, well before `phraseTimerId` — the melody clip is still genuinely queued when `handleTimeout` runs. Ran the test: passes. I also reasoned through the regression case without touching code (this is a read-only review): if the `queuedClips.forEach(...)` block were removed from `handleTimeout`, `shortEvents.find(e => e.eventName === 'clip_unqueued')` would return `undefined`, failing `expect(undefined).toMatchObject(...)`, and `getQueuedClips()` would still show the melody pillar occupied, failing the second assertion too. The test would catch this regression on two independent assertions.

**2. The updated (not just added) test — a legitimate strengthening.**
`git show 6806812 -- sim/test/simulator.test.ts` and a direct read of the parent commit's file confirm the pre-PR version of this exact test contained `expect(eventNames()).not.toContain('master-key_changed');` with the comment "The real handleTimeout clears the master key without emitting" — i.e., it asserted the bug. The PR flips this to `expect(eventNames()).toEqual([..., 'master-key_changed'])` plus `expect(lastEvent('master-key_changed')?.data).toEqual({ key: '' })`. This is strictly more specific than the old assertion and fails if the fix is reverted — a real strengthening, not a weakening dressed up as one.

**3. Backend/sim parity — confirmed equivalent for the targeted fix.**
Read both `handleTimeout` implementations side by side (`backend/adapter/AbletonAdapter.ts:148-163`, `sim/core/simulator.ts:170-185`). Both, after stopping playback: set the key to `''`, emit `master-key_changed` with the cleared key via the without-reset variant, then walk `queuedClips` and for every occupied pillar null the slot and emit `clip_unqueued` with `{ ...queuedClip, clip: undefined }` via the without-reset variant — matching order, matching payload shape, matching the existing `stopOrRemoveClipFromQueue` queued-removal payload convention in both files. The two files' pre-existing structural differences (backend stops tracks via awaited Ableton commands and relies on the async `playing_slot_index` listener to actually emit each `clip_stopped`; the sim does it synchronously and directly) are untouched by this diff and orthogonal to what WOW-018 fixes — not a parity gap introduced here.

**4. Without-reset emit variant claim — verified accurate, not just plausible.**
`backend/event/OutgoingEvents.ts:23-26`: `emitEvent` calls `AbletonAdapter.restartTimeoutTimer()` unconditionally before emitting; `emitEventWithoutResetingTimout` (`:19-21`) does not. `restartTimeoutTimer` (`AbletonAdapter.ts:187-192`) clears and fully re-arms both timeout timers. Had `handleTimeout` used `emitEvent`, its own cleanup would re-arm a fresh idle window immediately after the timeout it's handling — exactly the claim. Sim's `emit(eventName, data, resetTimeout = true)` (`simulator.ts:132-136`) mirrors this, and `handleTimeout` explicitly passes `false` for all three of its emissions. Confirmed accurate and consistent in both files.

## Nits (non-blocking)

1. **Backend: dropped queued clips at timeout don't get their `pitch_coarse` reset, unlike the established pattern.** `stopOrRemoveClipFromQueue`'s queued-removal branch (`AbletonAdapter.ts:340-346`) resets a dequeued clip's transposition via `FindAllClipsInLoop` + `transposeClipToNewKey(..., '')` when `keyLockEnabled`. The new `handleTimeout` queued-clip loop (`:155-162`) does not. Likely benign — `queueClip` unconditionally re-transposes every clip in the loop to the current `masterKey` the next time it's queued (`:257-260`), so any stale `pitch_coarse` self-heals on next use — but it's a genuine asymmetry with the existing pattern that neither the PR body nor the agent-note calls out, and it's outside what the sim can model or a vitest test can verify (Ableton pitch state). Worth a one-line confirmation from audio-ableton-reviewer that a stopped, never-replayed clip sitting at a non-zero `pitch_coarse` until its next queue is a non-issue.
2. **Sim: `handleTimeout` doesn't clear `phraseTimerId`.** Pre-existing, untouched by this diff. If a clip is still queued when timeout fires, its `phraseTimerId` (armed by the earlier `queueClip` call) keeps running; when it eventually fires, `triggerQueuedClips`'s `if (!queued) return` guard makes it a no-op since `queuedClips[pillar]` is already null by then — confirmed harmless in the current code, including in the new test's own timeline. Flagging only because I traced the full timer mechanics for point 1; not a WOW-018 regression and not blocking.
