2. **`TIMEOUT_IN_MILISECONDS`/`TIMEOUT_WARNING_IN_MILISECONDS` → `TIMEOUT_IN_MILLISECONDS`/`TIMEOUT_WARNING_IN_MILLISECONDS`, `emitEventWithoutResetingTimout` → `emitEventWithoutResettingTimeout`**: renamed in `backend/adapter/AbletonAdapter.ts` (definitions, all internal uses, and the exported-object keys) and `backend/event/OutgoingEvents.ts` (definition + export). 3. **`console.log('clips', clips)` removed** (`AbletonAdapter.ts`, inside `queueClip`) — deleted outright rather than converting to `Logger.debug`: `clips` is an array of raw `ableton-js` `Clip` objects, not meaningful structured log data (no clean serialization, not human-readable in a pino log line), so a debug-level dump wouldn't have added real diagnostic value.
4. **Deleted `KEY_LEADER_ORDER` and `ATTRACTOR_STATE_CLIP_NAME`** (declarations + their entries in the exported `AbletonAdapter` object). Grepped the whole repo (all `.ts`/`.tsx`, including every test file) — confirmed genuinely never read anywhere, not even via the exported object. Chose delete over "ticket the intended feature": both names strongly suggest a half-considered feature (`KEY_LEADER_ORDER`'s Vox-first ordering is the exact reverse of `TRIGGER_ORDER`'s Drums-first ordering, suggesting a "let Vox lead key/pitch decisions separately from playback-trigger order" idea; `ATTRACTOR_STATE_CLIP_NAME = 'Wicked Casting'` suggests an "idle/attract-mode clip" idea) — but I have no way to confirm what was actually intended, and inventing a speculative feature ticket from a guess would be worse than just noting the guess here and letting git history preserve the original code if anyone wants to revisit it.
5. **`[1, 2, 3, 4]?.map` → `[1, 2, 3, 4].map`** (`CurrentlyPlayingListContainer.tsx`) — the array literal is never null/undefined, so `?.` was a no-op.
6. **Removed the dead `info?.clipName?.trimStart()` initial assignment**, replaced with a plain `let clipName = '';`. The value was unconditionally overwritten by the if/else immediately below either way, so the `.trimStart()` computation's result was never used. Per the ticket's own instruction, confirmed the else-branch's empty-string behavior is intended and added a comment: displaying nothing (rather than falling back to `info?.clipName`, an internal CSV/Ableton identifier like "Flashback Drums 10A 135") when artist/songTitle metadata is missing is more consistent with the visitor-facing "Artist - Song Title" format used in the happy path than showing a raw internal name would be. **Zero behavior change** — this is documenting the existing, unchanged behavior, not altering it.
7. **`ingredients_contianer` → `ingredients_container`** (`RecipeBoxContainer.tsx`) — grepped for any other reference to the misspelled id (CSS, tests, JS query selectors); none found, purely cosmetic.
8. **Dynamic Tailwind class replaced with a conditional between two full literal strings** (`CurrentlyPlayingListContainer.tsx`, the `id='bpm'` div): `` `col-start-${index % 2 === 0 ? 2 : 1} col-span-3 font-fondamento` `` → `index % 2 === 0 ? 'col-start-2 col-span-3 font-fondamento' : 'col-start-1 col-span-3 font-fondamento'`. Produces byte-identical class strings to before (verified live, see below) — the fix is about making both literals independently discoverable by Tailwind's static scanner, not changing output. Matches the pattern the same file already uses two other places (`index % 2 === 0 ? (<div>...) : null` and the two `col-start-2`/`col-start-1` blocks further down).
9. **Deleted the commented-out `enrichRecommendations` block** (`MusicDatabaseService.ts`), replaced with a short comment documenting the intentional frontend/backend divergence. Verified both halves of the claim directly: `src/util/ClipDatabaseUtil.ts:11` does call `CsvUtil.enrichRecommendations` (frontend runs it); `sim/test/music-database.test.ts:33` asserts `does not attach recommendedClips — EnrichRecommendations is disabled in the real backend` (matching the backend's now-commentless omission).
10. **`sim/server.ts`'s stale pre-WOW-011 path comments updated** (4 of the ticket's cited 5 locations actually contained a stale path — `backend/events/incoming-events.ts` / `backend/ableton-api.ts` — the 5th cited line, `:4`, already correctly said `backend/index.ts`, nothing to fix there) to the current `backend/event/IncomingEvents.ts` / `backend/adapter/AbletonAdapter.ts` paths.
