## The `phraseLeader` guard restructure — verified equivalent for the defined case

`AbletonAdapter.ts:213-229`. Old: bare `if (playingClip.clipName.replace(...) === phraseLeader.clipName.replace(...)) {...}`. New: `if (!phraseLeader) { Logger.debug(...); } else if (playingClip.clipName.replace(...) === phraseLeader.clipName.replace(...)) {...same body...}`.

- **`phraseLeader` defined (the only case that ever ran on `main`):** `!phraseLeader` evaluates `false`; control falls into the `else if`, whose condition is character-for-character the same expression `main` used, guarding the character-for-character same body (`PhraseLeaderService.findNextPhraseLeader` → `addPhraseLeader`/`triggerQueuedClips`, now with `.catch()`). Zero behavioral difference.
- **`phraseLeader` undefined (the ticket's named concrete crash trigger):** `phraseLeader` is declared `let phraseLeader: ClipInfo;` at module scope (`:33`) with no initializer and is assigned exactly once, at `addPhraseLeader`'s first line (`:261`) — never reset to falsy anywhere else in the file (confirmed via `grep -n phraseLeader` against both `main` and the PR head: 3 references in `main`, same 3 plus the new guard in the PR). So this branch can only ever fire during the narrow startup window before the very first `playing_slot_index` event promotes a phrase leader — after that, for the remaining life of the process, `!phraseLeader` is always `false` and behavior is permanently identical to `main`. This confirms the guard's blast radius is exactly the one edge case the ticket names, not a general behavior change.
- **Does the guard skip code that used to run?** No. `stopOrRemoveClipFromQueue` is `async`; a `throw` deep inside an `async` function body never propagates synchronously to its caller (ES2017 semantics — it rejects the returned Promise instead). On `main` this call was unawaited/uncaught at its only call site (`IncomingEvents.ts:90`, pre-PR), so the throw was fatal to the whole process, not just to statements after it — literally nothing later in the function ran either. Both old (crash) and new (guarded) behavior skip the exact same statements for this one invocation; the only difference is the new version's process survives. I independently read the full function (`AbletonAdapter.ts:196-257`) to confirm `isClipPlaying`/`isClipQueued` are `const`s captured at function entry (`:201-202`, `:233`), so the fact that execution now _reaches_ the `isClipQueued` block (unreachable pre-fix due to the crash) cannot fire an extra Ableton command in this scenario — matches audio-ableton-reviewer's independently-traced conclusion in their note's section 2 and informational observation 1. Worth a human's attention as documented reasoning, not as a defect.

## Type change — `cleanUpPhraseLeaderEventListener` (`AbletonAdapter.ts:34`)

`(() => unknown) | undefined` → `(() => Promise<unknown>) | undefined`. Verified directly against `backend/node_modules/ableton-js/ns/index.d.ts:15`: `addListener<T>(...): Promise<() => Promise<boolean | undefined>>` — the resolved cleanup function's own return type really is `Promise<boolean | undefined>`, which is assignable to the new `Promise<unknown>` annotation. The old annotation was simply wrong (a function that in fact always returns a `Promise`, mistyped as returning bare `unknown`). This is a pure compile-time annotation change — TypeScript types are erased at runtime, so this has no runtime effect regardless of what the annotation says; confirmed no cast, coercion, or new runtime branch accompanies it. `tsc` forced this fix the moment `.catch()` was added to a call typed as returning `unknown`, which is itself circumstantial confirmation the old type was inaccurate (if it had been accurate, `.catch()` would never have compiled against real usage).

### Should-fix

1. **`backend/adapter/AbletonAdapter.ts:215`** — the new `phraseLeader`-undefined message uses `Logger.debug(...)`, but `backend/util/Logger.ts` configures pino at `level: 'info'`, which suppresses `debug`. This means the message will **not** appear in normal operation — which directly undermines this same PR's own "How to verify" step 1 (both in the PR body and the handoff note), which tells the human verifier to look for exactly this debug line ("it should log a debug line ('No phrase leader set yet...')"). As written, that verification step won't produce visible output. Also flagged independently by Copilot (`AbletonAdapter.ts:216` thread). Suggested fix: either log at `.info(...)` (matching this file's dominant convention for lifecycle-adjacent messages — `docs/CODING_GUIDELINES.md`'s logging taxonomy doesn't define a use for `debug` at all, only trace/info/warn/error) or update the verification steps to say the line won't be visible at the default level.
2. **`backend/index.ts:12-20,49-52`** — `process.exit(1)` called on the same tick as `Logger.error(...)` in all three new exit paths (`unhandledRejection`, `uncaughtException`, `main().catch`). Also flagged by Copilot (`index.ts:20` thread). Checked `backend/util/Logger.ts`: `pino({ level: 'info' })` with no `transport`/custom `destination` configured, which uses pino's default synchronous stdout writer — so in the common case this is very likely safe today. But it's not guaranteed (stdout redirected to certain destinations, or a future `transport` addition, can make the write asynchronous), and this is exactly the crash-diagnostic path the ticket exists to make reliable. Suggested fix (fast follow, not blocking this PR): use `pino.final()` or a short explicit flush/tick before `process.exit()`, consistent with pino's own documented guidance for exit handlers.
