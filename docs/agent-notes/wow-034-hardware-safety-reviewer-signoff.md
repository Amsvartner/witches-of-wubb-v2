## 1. Can the new code hang `process.exit(1)` past the ~1500ms bound, under any state the crash left the process in?

**No, in every scenario traced below â€” but for a slightly different reason than the implementer's note gives, which I want to correct on the record.**

The bound is enforced at two independent layers:

- **Inner layer** (`AbletonAdapter.ts:65-85`, `stopAllClipsBestEffort`): guards `tracks` being populated (`:66 if (!tracks?.length) return;` â€” `tracks` is a module-level `let tracks: Track[]` (`:32`) only ever assigned inside `getTracksAndClips()` (`:320-430`, `tracks = await ableton.song.get('tracks')` at `:324`); before that resolves, `tracks` is `undefined`, so `tracks?.length` is `undefined`, the guard fires, and the function returns immediately with a log line, no attempt made). If `tracks` is populated, it fires `stop_all_clips` on `tracks.slice(0, 4)` in parallel (`Promise.all` over `.map`), each call individually `.catch()`-guarded, the whole body wrapped in its own `try/catch`.
- **Outer layer** (`backend/index.ts:28-31`, inside `crashExit`): `await Promise.race([AbletonAdapter.stopAllClipsBestEffort(), new Promise((resolve) => setTimeout(resolve, CRASH_EXIT_STOP_TIMEOUT_MS))])` with `CRASH_EXIT_STOP_TIMEOUT_MS = 1500` (`:15`), then unconditionally `process.exit(1)` (`:32`).

The critical property is that **the outer layer's bound does not depend on the inner layer behaving correctly.** `Promise.race` settles as soon as either promise settles; the `setTimeout(resolve, 1500)` timer is a bare, `ableton-js`-independent JS timer, so even if `stopAllClipsBestEffort()` never resolves at all, `crashExit` still proceeds to `process.exit(1)` at the 1500ms mark, abandoning whatever's still pending inside `stopAllClipsBestEffort()`. This matters because I checked the implementer's specific claim â€” "`sendCommand(command, timeout = 2000)` has a built-in 2000ms default timeout... an individual command cannot hang indefinitely even in the worst case" (`docs/agent-notes/wow-034-creative-tech-integrator-crash-exit-stop-clips.md`, investigation point 1) â€” against the actual installed source (`backend/node_modules/ableton-js/index.js`, confirmed `3.1.5` via `package.json` and matching `backend/yarn.lock:125`), and **this claim is not quite right**:

- `Ableton.prototype.sendCommand` (`index.js:451-487`) does register a `setTimeout(..., timeout)` (default 2000, confirmed at `:452`) that rejects with `TimeoutError` if no response arrives â€” this part is accurate for the common case.
- But `Ableton.prototype.handleDisconnect` (`index.js:150-160`), which fires on a heartbeat-detected disconnect or an explicit disconnect message from the Ableton-side script, does this to every in-flight command: `this.msgMap.forEach((msg) => msg.clearTimeout()); this.msgMap.clear();` â€” it **cancels each pending command's timeout and drops it from the map, but never calls its `rej(...)`.** A command that is in-flight (sent, awaiting a response) at the exact moment a disconnect event fires therefore has its promise **permanently stuck pending â€” it will never resolve or reject.** This is a real bug/gap in `ableton-js@3.1.5` itself, not something in this repo's code, and not something the implementer's investigation checked (their point 1 covers the timeout path only, not the disconnect path).
- Applied to this PR: if Ableton disconnects while `stopAllClipsBestEffort()`'s four freshly-issued `stop_all_clips` calls are still in flight, all four of those promises could hang forever at the `ableton-js` layer, `Promise.all(...)` inside `stopAllClipsBestEffort` would never settle, and `stopAllClipsBestEffort()`'s own returned promise would never resolve.
- **This still does not break the bound**, because `crashExit`'s outer `Promise.race` doesn't care whether `stopAllClipsBestEffort()` ever settles â€” the bare 1500ms timer resolves regardless and `process.exit(1)` fires on schedule. So the actual safety property holds, but it holds _because of the outer race against an independent timer_, not because "an individual command cannot hang indefinitely" â€” that inner claim is false in the disconnect-mid-flight case, and the code is safe in spite of it, not because of it. I'd call this a minor, non-blocking finding: correct the code comment/rationale (`index.ts:12-14`'s comment and the agent-note) to stop asserting `ableton-js`'s own timeout is what bounds this, since it isn't, reliably; the real guarantee is the outer race. This is a documentation-accuracy point, not a code defect â€” the shipped code is already written the safer way (racing against an independent timer rather than trusting the dependency), which is the right call whether or not the implementer's stated reasoning for it was fully accurate.

Other states traced, all bounded:

- **`tracks` partially populated** (e.g., fewer than 4 elements, if `getTracksAndClips` crashed mid-loop after assigning `tracks` but before finishing per-track setup): `.slice(0, 4)` and `.map()` handle a short array fine â€” attempts whatever exists, no hang, matches the same `slice(0, 4)` convention already used by `getTrackVolumes` (`AbletonAdapter.ts:448`) and the hardcoded `4` in `handleTimeout` (`:56`).
- **`sendCommand` throwing synchronously instead of rejecting**: traced the full call chain â€” `Track.sendCommand` is inherited, unmodified, from `Namespace.prototype.sendCommand` (`ns/index.js:97-103`), which forwards to `Ableton.prototype.sendCommand` (`index.js:451`). That function's entire body is `new Promise(function (res, rej) { ...; _this.sendRaw(msg); })`. `sendRaw` (`index.js:614-632`) itself can throw synchronously â€” `if (!this.client || !this.serverPort) throw new Error("The client hasn't been started yet...")` (`:615-617`) â€” but because this throw happens _inside_ a `Promise` executor, the `Promise` constructor automatically converts it into a rejection of that promise rather than letting it escape synchronously to the caller. So `track.sendCommand(...)` is guaranteed to always return a promise â€” it cannot throw synchronously to its caller under any code path I could find in this dependency, including the "client torn down" state (exactly the "the very state that caused the crash may make `tracks`/`ableton` unusable" risk I raised in the WOW-014 sign-off). That rejection is caught by the per-call `.catch((err) => Logger.error(...))` at `AbletonAdapter.ts:77-79`.
- **The `.map()` construction itself throwing** (e.g., a hole in the `tracks` array making `track` `undefined`, so `track.sendCommand` would be a `TypeError` thrown synchronously _during_ `Array.prototype.map`, before `Promise.all` is even called): this throw happens synchronously inside the `try` block at `AbletonAdapter.ts:70-81`, so it's caught by the `catch (err)` at `:82-84` â€” function still resolves normally, no throw escapes `stopAllClipsBestEffort()`.
- **`Logger.error` itself throwing** inside the per-call `.catch()` or the outer `catch`: traced this up through `crashExit` â€” if it happens inside the per-call `.catch()`, the resulting rejection is an element of the `Promise.all` array, caught by `stopAllClipsBestEffort`'s own outer `try/catch` (`:70-84`), so `stopAllClipsBestEffort()` still resolves. If it happens inside `stopAllClipsBestEffort`'s outer `catch` itself, the throw converts to a rejection of `stopAllClipsBestEffort()`'s own promise, which makes `Promise.race` in `crashExit` reject (not hang) â€” `await Promise.race(...)` then throws inside `crashExit`, skipping the `process.exit(1)` line, but that rejection propagates out of `crashExit(...)`'s own returned promise to the `.catch(() => process.exit(1))` backstop chained at each `process.on(...)` registration (`index.ts:36`, `:40`) â€” `process.exit(1)` still fires, in this case _faster_ than 1500ms, not slower.
- **`AbletonAdapter.stopAllClipsBestEffort` not being a function** (import/export wiring failure): would throw synchronously when `crashExit` constructs the `Promise.race([...])` array (`index.ts:29`); since this happens before `crashExit`'s first `await`, the synchronous throw is converted into a rejection of `crashExit`'s own returned promise (standard `async function` semantics), caught by the same `.catch(() => process.exit(1))` backstop. Confirmed moot anyway: `stopAllClipsBestEffort` is correctly added to the `AbletonAdapter` export object (`AbletonAdapter.ts:588`) alongside the pre-existing `handleTimeout`.
- **Event-loop delay pushing the 1500ms timer later**: nothing in this diff does long synchronous work â€” the `.map()` over 4 tracks, `zlib.deflateSync` on tiny JSON payloads inside `sendRaw`, and `dgram.send()` (non-blocking OS-level enqueue, no DNS lookup since the target is a literal `127.0.0.1`) are all effectively instantaneous. Not a realistic risk from this PR's own code.

One more thing worth stating as a positive finding, not just risk analysis: `sendRaw` calls `this.client.send(...)` _synchronously_, inside the same tick as the `sendCommand()` call, before any await. That means the actual UDP `stop_all_clips` packets for all 4 tracks are dispatched onto the wire almost immediately when `stopAllClipsBestEffort()` runs â€” well within the first few milliseconds of the crash, not after waiting up to 1500ms. Even in the worst case where the 1500ms bound is hit because Ableton is slow to acknowledge, the stop commands were very likely already sent and received by Ableton's side before the process exits; the bound only limits how long the backend waits around for _confirmation_, not how long it waits before _sending_. This makes "best-effort" a stronger guarantee in practice than the name implies.

## 2. Can the new code itself throw/reject in a way that prevents `process.exit(1)` from ever being called?

**No.** Covered in the traces above; every throw/reject path I could find terminates at one of: (a) `stopAllClipsBestEffort`'s own inner or outer `catch` (resolves normally), (b) `crashExit`'s `Promise.race` resolving via the timer branch regardless of the other branch's fate, or (c) the `.catch(() => process.exit(1))` backstop chained at both `process.on(...)` registrations (`index.ts:36`, `:40`), which exists specifically to catch anything unforeseen in `crashExit` itself. I could not construct a path that reaches neither `crashExit`'s own `process.exit(1)` call nor the backstop.

## 3. Re-entrancy: does `isCrashExiting` behave as claimed under a second crash mid-shutdown?

**Yes.** `isCrashExiting` (`index.ts:16`) is checked and set synchronously at the very top of `crashExit` (`:19-26`), with no `await` between the check and the set â€” since Node is single-threaded and there is no yield point in between, there is no interleaving window where two concurrent invocations could both observe `isCrashExiting === false`. A second `unhandledRejection`/`uncaughtException` firing while the first is mid-race (inside the `Promise.race` at `:28-31`) hits the early-return branch (`:19-24`): logs its own error (tagged "crash exit already in progress, skipping duplicate stop attempt" so the second failure isn't lost for postmortem) and returns â€” it does not call `AbletonAdapter.stopAllClipsBestEffort()` again, does not construct a second `Promise.race`, and does not call `process.exit(1)` itself. The _first_ invocation's `crashExit` call is still the only one that will call `process.exit(1)`, bounded to ~1500ms after the _first_ crash regardless of how many subsequent crashes pile up during that window. I do not see a scenario where this guard itself causes a hang or a missed exit â€” the guard only ever short-circuits to "log and return," never to "wait for something."

## 6. Judgment call: is this a net improvement, or a new risk category that should have gone to a human instead?

**Net improvement, and I don't think this should have been kicked to a Decision-needed entry.** My own reasoning, not a restatement of the implementer's:

A "Decision needed" entry is the right tool when the answer depends on product/creative/business judgment this reviewer role and this agent aren't positioned to make â€” volume ceilings, show-restart policy, what the LEDs should look like. This ticket isn't that shape. It's a bounded engineering question with a checkable answer: _given a fixed, versioned dependency and precise `Promise`/event-loop semantics, can a bounded best-effort stop attempt be added to a crash handler without risking a hang?_ I set the bound myself in the WOW-014 sign-off ("a bounded (e.g. 1-2s timeout, wrapped in its own try/catch, never delaying `process.exit(1)` beyond that bound)") specifically because I judged that constraint sufficient to make this safe to build without further human input, provided the implementation actually honored it. Having now independently traced the implementation against that constraint â€” including going one level deeper than the implementer's own investigation and finding a real gap in `ableton-js`'s disconnect handling â€” I'm satisfied it does honor the constraint, and for a slightly more robust reason (the outer race against an independent timer) than the implementer's note claims (reliance on `ableton-js`'s own per-command timeout, which I found is not actually watertight). That the code turned out to be safe for a better reason than stated is a good outcome, but it's also exactly why "verify independently, don't just trust the narrative" earns its keep here â€” had the implementation instead relied solely on `ableton-js`'s internal timeout (e.g., a single `await stopAllClipsBestEffort()` with no outer race), the disconnect-mid-flight gap I found would have been a genuine hang risk, and I would be writing a BLOCK verdict with a request to add exactly the outer race that's already present.

Is "await before `process.exit(1)` on a crash path" a new risk category in general? Yes, legitimately â€” that's exactly why I asked for it to be bounded rather than approving an open-ended version in the first place. This PR's answer to that risk is about as good as I think is achievable in this stack short of a process-supervisor-level solution: two independent bounds stacked (the dependency's own ~2000ms timeout as a soft inner bound, and a hard ~1500ms outer bound that doesn't trust the dependency to behave), every individual failure mode caught at the layer closest to where it can occur, and a final backstop (`â€‹.catch(() => process.exit(1))`) that doesn't trust the new `crashExit` function itself either. I would not have approved a version of this that awaited `stopAllClipsBestEffort()` directly without the outer race.

Two things keep this from being an unqualified "solved," both minor and non-blocking:

- **Minor:** the code comment (`index.ts:12-14`) and the agent-note's investigation point 1 assert that `ableton-js`'s own per-command timeout alone makes an individual command unable to hang indefinitely. Per section 1 above, that's not true in the disconnect-mid-flight case. Recommend correcting the comment to credit the outer `Promise.race` as the actual guarantee, so a future reader doesn't lean on the inner timeout alone if this code is ever refactored.
- **Minor:** `main()`'s startup-failure `.catch()` (`index.ts:70-73`) still never attempts a stop, including in the narrow case where `tracks` was already populated before a _later_ startup step (e.g. `getTrackVolumes()`, which runs after `getTracksAndClips()` inside `startAbleton()`) throws. This is unchanged from pre-PR behavior, explicitly out of this ticket's stated scope, and low real-world likelihood (no RFID tag can have been processed yet at that point in boot, since socket/OSC listeners aren't registered until after `startAbleton()` resolves) â€” but it is a real, narrow gap. Worth a follow-up note alongside WOW-032, which is already working the startup-timeout territory immediately adjacent to this.

Neither rises above minor; neither is a regression; neither blocks this PR.
